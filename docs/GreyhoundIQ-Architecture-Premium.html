0Scope of this document

This doc covers the four interlocking systems you asked for:

  1. Supabase backend as the canonical data store, auth provider, realtime channel, and object store
  2. Per-user agent memory so every agent remembers each customer and resumes conversations
  3. Hermes Pi CLI harness so agents run via your existing Hermes Agent CLI on the MiniMax M3 token plan
  4. Rich-media customer-to-customer messaging (text, image, video, voice)
  5. Production-grade security across every layer (the part that actually matters)

It deliberately does not cover the race-data display side of GreyhoundIQ (dogs, races, results) — that surface is already built. This doc covers everything new, including the data-ingestion architecture that feeds it (section 4) and the AI/DNA intelligence layer.

1Goals and non-goals

Goals

  • Each user has a persistent identity across web, future mobile, and AI agent channels.
  • Each user has a personal AI assistant that remembers prior conversations, preferences, owned dogs, watched races, and unfinished work.
  • Customers can message each other with text + image + video + voice, with realtime delivery and message persistence.
  • All data lives in Supabase (Postgres + Storage + Realtime + Auth).
  • Agents run via Hermes Agent CLI as subprocesses on your token plan — no separate LLM API costs.
  • Production security: real auth, real RLS, real threat model. Not toy.

Non-goals (ponytail)

  • Mobile apps (web-only for now).
  • End-to-end encryption of messages (Supabase TLS in transit, at-rest encrypted, but server can read — fine for greyhound community messaging).
  • Multi-region / global edge. Single-region Supabase project.
  • Voice/video calling (WebRTC). Voice messages (recorded audio attachments) are in scope.
  • Real-time AI agent-to-agent hand-offs. Single agents per task.

2System architecture overview

System architecture
Next.js 14 front end → service layer → three data/agent adapters → managed Supabase + Hermes infra

Key design choice: Prisma + Supabase Client both talk to the same Postgres database. Prisma owns complex queries (joins, aggregations, typed writes). Supabase client owns realtime subscriptions and storage uploads. Two clients, one DB, no sync issues.

4Data Ingestion Architecture

The race-data spine (dogs, races, results) was previously treated as "already built." This section makes the ingestion layer first-class: how every external feed is fetched, normalized, deduplicated, scheduled, and monitored into one canonical store the rest of the platform reads from. Every source is wrapped in its own adapter — the application code never talks to a vendor API directly.

Data ingestion architecture
Per-source adapters normalize every feed into one canonical Postgres store, on a cron/queue schedule

4.1 The adapter pattern

Each feed (Topaz, Betfair Hub, Betfair Exchange, GA stud book, Isolynx) implements a common SourceAdapter interface. This isolates vendor quirks — auth, pagination, rate limits, response shape — behind one contract, so adding or swapping a source never touches business logic.

interface SourceAdapter {
  readonly source: "topaz" | "betfair_hub" | "betfair_exchange" | "ga_studbook" | "isolynx";
  fetch(cursor?: string): Promise<RawBatch>;     // auth + paginate + rate-limit
  validate(raw: RawBatch): ValidationResult;        // schema + type checks
  normalize(raw: RawBatch): CanonicalRecord[];      // map to our schema
  dedupKey(record: CanonicalRecord): string;        // content hash / natural key
}

4.3 Normalization & identity resolution

The hard problem isn't fetching — it's reconciling the same dog, race, or sire across sources that use different IDs and naming. The normalize step resolves every record to a canonical entity:

  • Dogs: matched on (name + whelping date + sire/dam) with a fuzzy fallback; Topaz ID and GA stud-book ID stored as cross-references.
  • Races: keyed on (track + date + race number); odds from Betfair joined to the Topaz race via the same key.
  • Conflicts: source precedence is explicit — official results (Topaz) beat odds-derived inferences (Betfair); Isolynx times are stored separately and never overwrite official timing.

4.4 Ingestion data model

model IngestionRun {
  id           String   @id @default(cuid())
  source       String   // topaz | betfair_hub | betfair_exchange | ga_studbook | isolynx
  jobType      String   // live_fields | results | backfill | pedigree
  status       String   @default("running") // running | ok | partial | failed
  cursor       String?  // resume point for paginated/incremental pulls
  rowsFetched  Int      @default(0)
  rowsWritten  Int      @default(0)
  rowsRejected Int      @default(0)
  error        String?
  startedAt    DateTime @default(now())
  finishedAt   DateTime?

  @@index([source, jobType])
  @@index([status])
}

model SourceRecord {
  id           String   @id @default(cuid())
  source       String
  sourceId     String   // the vendor's native id
  entityType   String   // dog | race | result | sire | market
  entityId     String?  // resolved canonical id (null until matched)
  sha256       String   // content hash for change detection / dedup
  rawJson      String   // original payload, for replay + audit
  fetchedAt    DateTime @default(now())

  @@unique([source, sourceId, sha256])
  @@index([entityType, entityId])
}

Why keep raw payloads? SourceRecord.rawJson lets us re-run normalization after a mapping bug without re-hitting (rate-limited, licensed) APIs — and gives a full audit trail of exactly what each source returned.

4.5 Scheduling, monitoring & failure handling

ConcernApproach
SchedulingSystem cron / lightweight queue hitting internal endpoints with a shared-secret header (no Redis/BullMQ at this scale). Live jobs every 2 min on race days; off-peak idle.
IdempotencyEvery write keyed on (source, sourceId, sha256) — re-running a job never double-inserts.
Partial failureIngestionRun.status = partial with rowsRejected logged; bad rows quarantined, good rows committed.
Data-quality monitoringAccuracy dashboard compares our results against the source; alerts on row-count drift, schema changes (a vendor field disappearing), and Isolynx-vs-official discrepancies.
Source outageAdapters fail independently; a Topaz outage doesn't stop Betfair odds. Last-good data is served with a freshness timestamp.
Schema driftValidation step fails loudly on unexpected shapes (undocumented "beta" endpoints can change) → alert, not silent corruption.
Licensing gates ingestion, not just display
Several feeds (Topaz commercial redistribution, Isolynx, race-fields) require written agreements before any production ingestion for a commercial service. The adapter layer is built first against free/public sources (Betfair Hub) so engineering proceeds while licensing is negotiated in parallel — but the switch to licensed live feeds is a hard gate before public launch. See the business plan's Data Sources & Feed Strategy for the commercial/legal detail.

Every table gets RLS enabled. Sample policies:

-- USERS: self + others' public fields
alter table "User" enable row level security;

create policy "user_read_public" on "User" for select using (
  auth.uid() = "User"."supabaseUserId"
  or exists (
    select 1 from "Profile" p
    where p."userId" = "User".id and p."showOwnedDogs" = true
  )
);

create policy "user_update_self" on "User" for update using (
  auth.uid() = "User"."supabaseUserId"
);

-- MESSAGES: only sender + recipient can read
create policy "message_read_participants" on "Message" for select using (
  exists (
    select 1 from "Conversation" c
    where c.id = "Message"."conversationId"
    and (c."participantAId" = auth.uid()::text
      or c."participantBId" = auth.uid()::text)
  )
);

-- MEMORY: only owning user (and service role for agents)
create policy "memory_read_self" on "MemoryEntry" for select using (
  "userId" = (select id from "User" where "supabaseUserId" = auth.uid())
);
Service role key bypass
RLS is bypassed by the service role key. Used only in agent harness subprocesses, cron jobs, and admin tooling. Never exposed to the browser. Stored in env var only.

6Authentication

5.1 Strategy

Supabase Auth handles the heavy lifting. Used directly — no NextAuth wrapper, fewer abstractions, less attack surface.

  • Email + password: argon2id (Supabase default). Min 12 chars.
  • Magic link: opt-in, Resend SMTP.
  • OAuth: Google (P1), Apple (P2).
  • Session storage: localStorage for web. HTTP-only cookie via @supabase/ssr for SSR.
  • Refresh tokens: rotation on every API call.

5.2 Protected routes (SSR)

// src/lib/auth/server.ts
import { createServerClient } from "@supabase/ssr";
import { cookies } from "next/headers";

export async function getServerUser() {
  const cookieStore = await cookies();
  const supabase = createServerClient(/* ... */);
  const { data: { user } } = await supabase.auth.getUser();

  return prisma.user.findUnique({
    where: { supabaseUserId: user.id },

7.4 Cancellation

// src/lib/agents/cancel.ts
export async function cancelAgentRun(runId: string) {
  const run = await prisma.agentRun.findUnique({ where: { id: runId } });
  if (!run?.harnessPid) return;
  process.kill(run.harnessPid, "SIGTERM");
  setTimeout(() => {
    try { process.kill(run.harnessPid, "SIGKILL"); } catch {}
  }, 5000);
  await prisma.agentRun.update({
    where: { id: runId },
    data: { status: "cancelled", completedAt: new Date() },
  });
}

10AI & DNA Matching Engine

This section covers the two intelligence systems that differentiate GreyhoundIQ: the race-prediction model and the DNA / genetic-matching engine (Testmate). Both run server-side, both write back into the data model, and both are gated by subscription tier.

9.1 Race-prediction model

A gradient-boosted model (XGBoost/LightGBM class) scores every runner in an upcoming race. It is trained offline on 5+ years of unified national form and Betfair data, versioned, and served behind an internal endpoint — never recomputed per request from scratch.

  • Features: box, distance, track, going, recent sectionals, trainer strike rate, days since last run, weight trend, class.
  • Outputs: calibrated win probability, predicted finishing position, confidence score, and per-runner factor attributions (for explainability).
  • Serving: batch-scored when fields are declared, cached in RacePrediction, invalidated if fields change.
  • Tiering: full predictions + confidence on Pro+; teaser (top pick only) on Pro.

9.2 DNA & genetic-matching engine (Testmate)

Given any sire × dam, the engine builds a joint 5-generation pedigree, computes an inbreeding coefficient, scans for recessive-carrier overlap, and runs a Monte-Carlo simulation to project litter performance.

DNA and genetic-matching pipeline
From a sire × dam request to a full litter report — COI, risk flags, and Monte-Carlo trait projection

9.3 Inbreeding coefficient (COI)

COI is computed with Wright's coefficient over the merged pedigree — the probability that two alleles at a locus are identical by descent. The result drives the colour-coded gauge: green (<5%), amber (5–10%), red (>10%). Calculations are memoised per sire×dam pair in DnaMatch so repeat lookups are instant.

9.4 Genetic-risk flags

Health gate, not a diagnosis
Recessive-risk flags indicate carrier overlap in the pedigree for known conditions — they are a breeding-decision aid, not a clinical test. The UI states this explicitly, and never presents a flag as a veterinary diagnosis.

15.1 Adopt now — the shortlist

Google TimesFM — form forecasting

A pretrained time-series foundation model (zero training for inference). GreyhoundIQ's data is deeply temporal — race times, sectionals, track-condition trends, a dog's form trajectory. Use TimesFM for form-trend forecasting and, crucially, as a cheap baseline to benchmark the ML win-probability model against. Open-source; runs on existing infra. Caveat: benchmark before trusting — "zero training ≠ zero setup."

DeepSeek-V4-Flash — agent cost lever

1M-token context at ~US$0.14/$0.28 per M (and $0.0028/M cache-hit) — a credible cheaper route or fallback for the per-user agents and the SEO generator alongside MiniMax M3. Structure prompts as fixed-prefix + variable-suffix to exploit cache-hit pricing across thousands of races. Gate first AU data-sovereignty review (Chinese provider) before any PII touches it.

Agent Skills + SkillSpector

Codify GreyhoundIQ's own conventions (Prisma/Supabase schema changes, App Router pages, Topaz/Betfair ingestion adapters) as version-controlled .claude/skills/ so Hermes/Claude Code-assisted work is reproducible. Gate every third-party skill through NVIDIA SkillSpector in CI to neutralise prompt-injection. Near-free; compounding dev-velocity return.

Hermes multi-agent crew pattern

Already our stack. The reference build (orchestrator + specialists, SQLite state, dashboard, per-agent memory on a VPS) is the blueprint for the per-user agents and an internal ops crew — Form Analyst, Breeding Advisor, SEO Writer. Heed the limitation: Hermes profiles can't hot-swap at runtime, so model per-user agents as personas, not swappable profiles.

Fal AI media pipeline (via Claude Code skill)

Terminal-orchestrated image/video editing (Nano Banana ~$0.08/img, Kling relight ~$0.17/sec) — the closest match to our Hermes CLI pattern. Concrete uses: polish founder/trainer update videos, standardise community thumbnails, background-swap user dog photos for marketing. Start with a budget-capped sandbox; audit the skill before install.

Crabbox — agent dev sandboxes

Disposable cloud sandboxes for AI coding agents; explicitly supports Hetzner + local Supabase. Let Hermes implement a feature, spin an ephemeral box, run tests + a Playwright capture of the UI, attach the recording to a PR for review — then tear it down. Free (you own the compute). Not a hostile-tenant security boundary — never pass secrets on the CLI.

Governance rules that shape the build

From the "20 AI Concepts" literacy review, four rules are now architecture constraints: (1) fine-tuning shapes tone, not facts — ground agents in Supabase retrieval (RAG), never fine-tune changing race data; (2) RAG doesn't eliminate hallucination — a bad chunk = a confident wrong "will dog X win?"; (3) prototype agents as deterministic workflows first to protect the MiniMax token budget; (4) treat all user-generated community content as untrusted input (prompt-injection surface). Pair with Claude Code ≥ v2.1.183 managed settings + branch protection + a read-only DB role for any agent before auto-mode work near production.

15.5 Full-corpus additions — what the deep read surfaced

Reading all 350 articles (not just the first 36) surfaced several higher-value, lower-risk additions that now lead the adoption plan. They cluster on the one gap the corpus agreed GreyhoundIQ has: per-user agent memory, plus the safety and evaluation scaffolding to run agents responsibly.

Mem0 on Supabase pgvector

The drop-in per-user agent memory layer (Apache-2.0) — persist each punter's tracks, followed dogs and staking style, and each breeder's priority markers, in the Postgres you already run. The single highest value-to-effort move in the corpus. (Hermes + Honcho is the alternative; Honcho is AGPL-3.0 — isolate it.)

Prompt-eval pipeline — race results as ground truth

Greyhound racing gives free, automatic ground truth: past results. Git-version the win-probability prompts, score accuracy & calibration against a held-out race set on every change, and block deploys that regress >5%. The cheapest safeguard for a product people bet on.

Agent sandboxing + approval gates

Default-deny Docker sandboxes, read-only data mounts, and tiered human approval on every Supabase mutation / Betfair-credential / money-adjacent action. The corpus repeatedly cites the OpenClaw one-click RCE (CVE-2026-25253) — this is the price of agent autonomy.

n8n + Firecrawl ingestion

Self-hosted n8n orchestrates the Topaz/Betfair pulls into Supabase (schedule, classify, route, alert on recessive-risk flags); Firecrawl-via-MCP fills the gaps Topaz/Betfair don't reach (stewards' reports, scratchings, track notes). Both self-host on Hetzner to keep data sovereign.

Cognee knowledge graph (breeding)

Model sire/dam/offspring/kennel/recessive-carrier as graph nodes for multi-hop DNA queries vector search can't answer — e.g. "which available studs share no recessive-risk carriers with this bitch's line?" Pairs with Testmate + the COI feature; runs on pgvector.

LocalAI + Whisper/Piper on Hetzner

One move, two wins: a data-sovereignty path (embeddings + a fallback LLM on-box, AU data never leaves) and the voice-messaging feature (transcribe voice notes, auto-captions, narrated form guides) without shipping user audio off-platform.

The corpus's one-line conclusion

The 350-article read quietly validated the stack (MiniMax M3 leads open-weight; Hermes-on-Hetzner is a first-class pattern; Supabase+pgvector is enough) and pointed at a single missing layer — memory that makes the agents remember each user — plus the security and evaluation discipline to run them safely. Build those, and the incumbents' read-only products have no answer.
