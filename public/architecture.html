0Scope of this document

This doc covers the four interlocking systems you asked for:

  1. Supabase backend as the canonical data store, auth provider, realtime channel, and object store
  2. Per-user agent memory so every agent remembers each customer and resumes conversations
  3. Hermes Pi CLI harness so agents run via your existing Hermes Agent CLI on the MiniMax M3 token plan
  4. Rich-media customer-to-customer messaging (text, image, video, voice)
  5. Production-grade security across every layer (the part that actually matters)

It deliberately does not cover the race-data side of GreyhoundIQ (dogs, races, results) — that's already built and approved. This doc is everything new.

1Goals and non-goals

Goals

  • Each user has a persistent identity across web, future mobile, and AI agent channels.
  • Each user has a personal AI assistant that remembers prior conversations, preferences, owned dogs, watched races, and unfinished work.
  • Customers can message each other with text + image + video + voice, with realtime delivery and message persistence.
  • All data lives in Supabase (Postgres + Storage + Realtime + Auth).
  • Agents run via Hermes Agent CLI as subprocesses on your token plan — no separate LLM API costs.
  • Production security: real auth, real RLS, real threat model. Not toy.

Non-goals (ponytail)

  • Mobile apps (web-only for now).
  • End-to-end encryption of messages (Supabase TLS in transit, at-rest encrypted, but server can read — fine for greyhound community messaging).
  • Multi-region / global edge. Single-region Supabase project.
  • Voice/video calling (WebRTC). Voice messages (recorded audio attachments) are in scope.
  • Real-time AI agent-to-agent hand-offs. Single agents per task.

Key design choice: Prisma + Supabase Client both talk to the same Postgres database. Prisma owns complex queries (joins, aggregations, typed writes). Supabase client owns realtime subscriptions and storage uploads. Two clients, one DB, no sync issues.

3.4 Agent memory (Cognee-style)

The shape is deliberately simple — no graph DB, no embeddings to start. Add embeddings later when scale justifies it.

model MemoryEntry {
  id            String  @id @default(cuid())
  userId        String
  user          User    @relation(fields: [userId], references: [id], onDelete: Cascade)
  kind          String  // episodic | semantic | preference | unfinished
  content       String
  source        String  // conversation | explicit_user_input | agent_inference | tool_observation
  sourceRef     String?
  importance    Float   @default(0.5)
  lastAccessedAt DateTime @default(now())
  accessCount   Int     @default(0)
  embedding     Unsupported("vector(1536)")?  // pgvector, Phase 2
  supersededById String?
  supersededAt   DateTime?
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  @@index([userId, kind])
  @@index([userId, importance])
}

model ConversationContext {
  id              String  @id @default(cuid())
  userId          String
  agentType       String  // race_analyst | breeding_advisor | form_reader | general
  lastMessageAt   DateTime @default(now())
  pendingAction   String? // JSON: { type: "wait_for_user_input", prompt: "..." }
  windowSize      Int     @default(50)
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  @@unique([userId, agentType])
}

model AgentRun {
  id               String  @id @default(cuid())
  userId           String?
  user             User?   @relation(fields: [userId], references: [id])
  agentType        String
  inputJson        String
  outputJson       String?
  toolInvocations  String?
  createdMemoryIds String?
  status           String  @default("pending")
  error            String?
  promptTokens     Int?
  completionTokens Int?
  durationMs       Int?
  harnessPid       Int?
  harnessSessionId String?
  createdAt        DateTime @default(now())
  completedAt      DateTime?

  @@index([agentType])
  @@index([userId])
  @@index([status])
}

Memory retrieval (Phase 1, no embeddings):

  1. Always include user's identity (display name, owned dogs, tier) — top 20 high-importance memories.
  2. Always include last ConversationContext.pendingAction.
  3. Always include last 5 episodic memories (conversation summaries).
  4. Skip semantic search until Phase 2 (pgvector).

  • Email: Custom SMTP via Resend.

  • pgcrypto (default — gen_random_uuid())

4.3 Row Level Security — the security backbone

Every table gets RLS enabled. Sample policies:

-- USERS: self + others' public fields
alter table "User" enable row level security;

create policy "user_read_public" on "User" for select using (
  auth.uid() = "User"."supabaseUserId"
  or exists (
    select 1 from "Profile" p
    where p."userId" = "User".id and p."showOwnedDogs" = true
  )
);

create policy "user_update_self" on "User" for update using (
  auth.uid() = "User"."supabaseUserId"
);

-- MESSAGES: only sender + recipient can read
create policy "message_read_participants" on "Message" for select using (
  exists (
    select 1 from "Conversation" c
    where c.id = "Message"."conversationId"
    and (c."participantAId" = auth.uid()::text
      or c."participantBId" = auth.uid()::text)
  )
);

-- MEMORY: only owning user (and service role for agents)
create policy "memory_read_self" on "MemoryEntry" for select using (
  "userId" = (select id from "User" where "supabaseUserId" = auth.uid())
);
Service role key bypass
RLS is bypassed by the service role key. Used only in agent harness subprocesses, cron jobs, and admin tooling. Never exposed to the browser. Stored in env var only.

5Authentication

5.1 Strategy

Supabase Auth handles the heavy lifting. Used directly — no NextAuth wrapper, fewer abstractions, less attack surface.

  • Email + password: argon2id (Supabase default). Min 12 chars.
  • Magic link: opt-in, Resend SMTP.
  • OAuth: Google (P1), Apple (P2).
  • Session storage: localStorage for web. HTTP-only cookie via @supabase/ssr for SSR.
  • Refresh tokens: rotation on every API call.

5.2 Protected routes (SSR)

// src/lib/auth/server.ts
import { createServerClient } from "@supabase/ssr";
import { cookies } from "next/headers";

export async function getServerUser() {
  const cookieStore = await cookies();
  const supabase = createServerClient(/* ... */);
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return null;
  return prisma.user.findUnique({
    where: { supabaseUserId: user.id },
    include: { profile: true },
  });
}

7Hermes Pi CLI harness

7.1 Why Hermes Agent CLI?

  • No separate LLM API costs (already on MiniMax M3 plan).
  • Same agent logic across web UI, future CLI, cron jobs.
  • Consistent tool calling format.

7.2 Architecture

┌──────────────────────────────────────────────────────────┐
│ Next.js API route: POST /api/agents/:type/run            │
│                                                          │
│ 1. Validates user + creates AgentRun row (status=pending)│
│ 2. Loads memory + builds system prompt                    │
│ 3. spawns: hermes agent --prompt  \              │
│               --tools ./src/lib/agents/tools/*.ts \      │
│               --output-format json                       │
│ 4. Streams output to a per-run temp file                 │
│ 5. Polls subprocess every 500ms (or uses stdio pipe)     │
│ 6. Updates AgentRun row with output + status             │
│ 7. Writes memory entries via memory.save tool            │
└──────────────────────────────────────────────────────────┘

7.3 Security at the harness boundary

ThreatMitigation
Prompt injection via user inputAgents use structured tool calls, not raw SQL.
Tool calling unauthorised APIsTools are whitelisted in agent config. Adding a tool requires code review.
Subprocess escapeRun as a dedicated low-privilege user. No shell. spawn('hermes', args, { shell: false }).
Token exfiltration via responsePost-process responses to strip credentials, secrets, foreign user IDs.
Runaway costHard limit: 50k tokens per run, 5 min duration, 10 runs/hour for Pro, 100/hour for Pro+.
Cross-user data leakageEvery tool that reads user data filters by currentUser.id. Enforced by a linter rule.

7.4 Cancellation

// src/lib/agents/cancel.ts
export async function cancelAgentRun(runId: string) {
  const run = await prisma.agentRun.findUnique({ where: { id: runId } });
  if (!run?.harnessPid) return;
  process.kill(run.harnessPid, "SIGTERM");
  setTimeout(() => {
    try { process.kill(run.harnessPid, "SIGKILL"); } catch {}
  }, 5000);
  await prisma.agentRun.update({
    where: { id: runId },
    data: { status: "cancelled", completedAt: new Date() },
  });
}

8.3 Voice and video

  • Voice: MediaRecorder API → opus/webm → same upload flow. Max 5 min.
  • Video: video/mp4 or video/webm. Max 60s, 200MB. Thumbnail at 1s mark via ffmpeg.wasm in Edge Function.
  • Inline preview: <video> with poster from thumbnail.
No E2E encryption in v1
For a greyhound community, E2E adds complexity without proportional value. Server can read messages (for moderation, search, agent memory). If users demand it later, add Signal Protocol library — but only after market validation.

9Realtime delivery

9.1 Delivery guarantees

  • At-least-once: server retries on client disconnect. Idempotency key prevents double-insert.
  • Order preservation: messages in a conversation always arrive in createdAt order.
  • Read receipts: receiver emits read event on message visible. Server updates readAt.

9.2 Presence and offline

Phase 2: presence channel per user. Phase 1: no presence. Offline messages persist in DB, delivered on reconnect via Realtime backlog or REST pagination with ?after=cursor.

11Privacy and compliance

11.1 Australian Privacy Principles (APPs)

  • APP 1 (accountability): privacy policy on site
  • APP 6 (use/disclosure): data only used for stated purposes
  • APP 8 (cross-border): Supabase region Sydney, no transfer
  • APP 11 (security): RLS, encryption, audit logs
  • APP 12 (access): users can export their data via /api/me/export
  • APP 13 (correction): profile edit form

11.2 Account deletion

Right to be forgotten. /api/me/delete:

  1. Soft-delete (mark User.isBanned=true, anonymize email).
  2. Wait 30 days (recovery window).
  3. Hard delete via scheduled job: anonymize PII in messages, posts, listings. Keep structural rows for analytics but no PII.

12Operational concerns

12.1 Background jobs

  • Memory decay (daily, 2am AEST)
  • Listing expiry (daily, 1am AEST)
  • Community moderator agent (every 30 min)
  • Agent run cleanup (hourly, kill runs > 24h)
  • Soft-delete hard-delete (daily)

Sentry. One project, three environments. User context attached via Sentry.user.id. PII stripped before sending.

13Build phasing

Each phase has a clear exit criterion. We stop and review between phases.

Phase 1: Foundation (Week 1–2)

  • Create Supabase project (Sydney region, Pro plan)
  • Configure RLS, storage buckets, realtime
  • Migrate Prisma schema to Postgres (section 3)
  • Wire Supabase Auth (email + Google)
  • Build the media upload pipeline (signed URLs, virus scan)
  • Build messaging backend (Prisma + Supabase Realtime)
  • Build minimal messaging UI
Exit: Two users can sign up, message each other with text + image.

Phase 2: Memory (Week 2–3)

  • Implement memory CRUD + decay
  • Implement conversation context
  • Build the agent harness (spawn Hermes Agent CLI)
  • Implement four core agents (race analyst, breeding advisor, form reader, moderator)
  • Build agent chat UI with memory display
Exit: User can have a multi-session conversation with the race analyst that picks up where they left off.

Phase 3: Community (Week 3–4)

  • Forum: categories, threads, posts
  • Marketplace: listings, listing media
  • Dog ownership claims
  • Reporting + moderation agent
Exit: A breeder can claim a dog, post in the forum, list a pup for sale, get notified of agent-moderated actions.

Phase 4: Polish (Week 4–5)

  • Voice + video messages
  • Audit logging
  • Data export + account deletion
  • Email digests
  • Documentation
  • External penetration test
Exit: Production launch.
